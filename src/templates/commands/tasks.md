# /tasks - 태스크 분해

계획을 실행 가능한 작은 태스크로 분해합니다.

## 사용법

```
/tasks
/tasks {User Story 번호}
```

예시:
```
/tasks            # 전체 태스크 생성
/tasks US1        # 특정 User Story만
```

## 프로세스

### 1. 입력 분석

다음 파일을 읽습니다:
- `specs/{current}/spec.md` - User Stories, 요구사항
- `specs/{current}/plan.md` - 기술 스택, 아키텍처

### 2. MVP First 전략

**핵심 원칙: 동작하는 최소 기능 먼저**

```
MVP 단계:
1. 핵심 기능만 구현 (Happy Path)
2. 에러 처리 최소화
3. UI 기본만 (스타일링 나중에)
4. 테스트는 핵심 경로만

확장 단계:
5. 에러 처리 보강
6. 엣지 케이스 대응
7. UI 개선
8. 테스트 커버리지 확대
```

**MVP 우선 이유:**
- 빠른 피드백 사이클
- 조기에 방향 확인 가능
- 불필요한 작업 방지

### 3. 태스크 ID 형식 (엄격)

```
T{3자리 숫자} [{우선순위}] [{User Story}]: {설명}
```

**형식 규칙:**
- `T001`, `T002`, ... (3자리 숫자, 001부터 시작)
- `[P1]`, `[P2]`, `[P3]` (우선순위, P1이 가장 높음)
- `[US1]`, `[US2]`, ... (관련 User Story)
- 설명은 **동사로 시작** (생성, 구현, 추가, 작성)

**예시:**
```
T001 [P1] [US1]: 로그인 폼 컴포넌트 생성
T002 [P1] [US1]: 로그인 API 호출 구현
T003 [P2] [US1]: 로그인 에러 처리 추가
T004 [P3] [US1]: 로그인 폼 스타일링
```

### 4. Phase 구분 (MVP 기반)

```
Phase 1: Setup (P1)
- 프로젝트 구조
- 의존성 설치

Phase 2: MVP Core (P1)
- 핵심 기능의 Happy Path
- 최소한의 UI
- 데이터 흐름 확립

Phase 3: Error Handling (P2)
- 에러 처리
- 유효성 검사
- 엣지 케이스

Phase 4: Polish (P3)
- UI 개선
- 성능 최적화
- 테스트 확대
```

### 5. 태스크 분해 원칙

**작은 단위로 분해:**
- 각 태스크는 **독립적으로 테스트 가능**해야 함
- 가능하면 **파일 1-2개만 수정**
- **명확한 완료 조건** 포함
- **30분~2시간** 내 완료 가능한 크기

**의존성 최소화:**
- 순환 의존성 금지
- 병렬 실행 가능한 태스크 표시

### 6. 파일 생성

`specs/{id}-{slug}/tasks.md` 생성 또는 업데이트
